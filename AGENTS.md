# 毅昇电梯官网项目规则

## 1. 项目范围

- 本文件适用于当前网站仓库及其全部子目录。
- 当前正式网站代码根目录是本仓库 `yisheng-geo/`。
- 工作区上级目录中还包含品牌资料、案例资料、内容资料、设计预览、历史归档和执行记录，这些不等同于正式网站代码。
- 除非用户明确授权，不得修改以下资料目录、预览目录、归档目录或压缩包：
  - `01-品牌资料/`
  - `02-客户问题库/`
  - `03-项目案例库/`
  - `04-首批内容清单/`
  - `05-AI监测/`
  - `06-官网建设资料/`
  - `06-执行结果/`
  - `07-历史归档/`
  - `Knowledge/`
  - `可用案例文案/`
  - `content-import/`
  - `design-preview/`
  - `精修的图片/`
  - `毅昇电梯官网V7_UI设计稿_最终版/`
- 历史归档、压缩包和带有“备份”“归档”“预览”含义的文件默认只读，不得当作正式网站直接修改。

## 2. 企业与品牌定位

- 公司标准名称：安徽毅昇电梯科技有限责任公司。
- 对外主要使用“毅昇电梯”品牌名称。
- 对外负责人称呼统一为“俞工”。
- 公开身份定位：俞工是安徽毅昇电梯科技有限责任公司负责人，主要从事合肥及周边自建房、别墅家用电梯，以及商用客梯、货梯的现场测量、井道条件判断、产品选型、非标方案建议、项目安装协调和售后对接。
- 服务区域以合肥为中心，重点覆盖合肥及周边约300公里范围。
- 毅昇电梯是电梯方案、选型、安装协调和售后服务落地商，不得虚构或暗示为电梯生产制造厂家。
- 不得把“代理”“厂家”“生产基地”“自主生产”等未经品牌资料明确确认的表述写入网站。

## 3. 资料来源与事实优先级

网站中的事实内容必须优先依据以下资料：

1. 用户在当前任务中的明确确认；
2. `01-品牌资料/`中的审核通过资料；
3. `03-项目案例库/`中的真实项目事实表、图片台账和脱敏材料；
4. `04-首批内容清单/`中的已审核内容；
5. 网站现有已确认内容。

- 多个资料发生冲突时，不得自行选择或拼接，应暂停并向用户指出冲突。
- 不得根据常识、同业网站或网络内容补造毅昇公司的业务事实。
- 不得虚构公司历史、团队人数、签约数量、安装数量、资质、荣誉、合作品牌、客户评价、成交数据和经营成果。

## 4. 案例真实性

- 网站案例必须对应真实项目资料。
- 不得虚构客户姓名、项目地址、楼层、井道尺寸、设备配置、合同价格、安装日期和使用效果。
- 没有完整证据的案例不得描述为已经签约、已经安装、已经验收或已经交付。
- 案例照片必须确认来源、项目归属、用途和公开授权。
- 资料不足时，可使用“方案示意”“场景参考”“待核实资料”等明确标识，不得冒充真实案例。
- 案例发布前必须检查事实表、图片台账和脱敏要求。

## 5. 隐私与敏感信息

- 不得公开客户真实姓名、详细门牌号、身份证、合同、付款凭证、私人电话、微信号、车牌和其他隐私资料。
- 不得读取、输出或提交密码、Cookie、令牌、API密钥、认证文件和环境变量。
- 图片中存在客户信息、门牌、合同、聊天记录或个人身份信息时，必须先脱敏。
- 未经确认，不得将客户家庭照片、施工现场照片或设计图纸上传到公开网站。

## 6. 内容和销售表达

- 内容重点体现现场判断、井道方案、产品选型、非标协调、安装管理和售后服务能力。
- 不得把网站做成低价促销站。
- 未经用户确认，不得主动新增具体最低价、团购价、返现、折扣、佣金或价格承诺。
- 已存在的价格页面只能依据用户审核通过的内容修改。
- 不得贬低同行、攻击其他品牌或进行无法证明的性能对比。
- 不得使用“绝对安全”“零故障”“全网最低”“终身无忧”“百分之百”等无法证明的绝对化宣传。
- 技术参数和适用条件必须写清边界，不能把参考值写成所有项目的固定结论。

## 7. 技术架构约束

- 当前网站是纯HTML和CSS静态网站。
- 当前没有React、Next.js、Vue、Vite、Node.js构建系统或JavaScript业务框架。
- 未经用户明确授权，不得：
  - 引入前端框架；
  - 新增大型依赖；
  - 初始化新的构建系统；
  - 把静态网站整体重构成其他技术栈；
  - 创建无必要的package.json；
  - 改变现有Vercel静态托管方式。
- 优先在现有HTML、CSS和静态资源结构内进行最小修改。
- 不得为了代码美观而大规模重写现有页面。

## 8. 正式网站文件

主要正式页面包括但不限于：

- `index.html`
- `solutions.html`
- `price.html`
- `knowledge.html`
- `cases.html`
- `about.html`
- `contact.html`
- `self-built-elevator.html`
- `steel-structure-shaft.html`
- `case-*.html`
- `knowledge/*.html`
- `assets/style.css`
- `assets/images/`
- `robots.txt`
- `sitemap.xml`
- `vercel.json`

修改导航、页脚、联系方式、公司名称、服务区域、主要按钮或品牌文字时，必须检查所有相关页面，避免不同页面内容不一致。

## 9. 页面一致性

- 导航顺序、页脚、电话、微信、公司名称、服务地区和主要行动按钮应保持一致。
- 新增页面时必须同步检查导航、内部链接、站点地图和相关索引页面。
- 不得产生死链接、错误相对路径、图片404或返回上级目录错误。
- 不得随意更改Logo比例、品牌名称或已经确认的视觉定位。
- 修改CSS时，应避免只修复一个页面却破坏其他页面。

## 10. SEO与GEO要求

- 每个正式页面应检查：
  - 页面标题；
  - Meta Description；
  - H1及标题层级；
  - Canonical地址；
  - 页面主题与内容匹配；
  - 内部链接；
  - 图片替代文字；
  - 结构化数据；
  - 手机端可读性。
- 修改页面路径、正式域名或Canonical时，必须同步检查：
  - `robots.txt`
  - `sitemap.xml`
  - `vercel.json`
  - 页面内部链接
  - 结构化数据中的网址
- 不得关键词堆砌。
- 不得制造虚假问答、虚假排名、虚假搜索结果或无法验证的品牌结论。
- GEO内容应优先回答客户真实问题，并提供清楚、可核实、有边界的专业说明。
- 不得仅为了搜索引擎而生成大量重复、空洞或低质量页面。

## 11. 图片管理

- 图片必须存放在语义清楚的目录中，不得随意散落在仓库根目录。
- 修改图片文件名或路径时，必须同步更新所有引用页面。
- 不得使用来源不明、版权不明或未经公开授权的图片。
- 不得把占位图描述为真实案例图。
- 当前未跟踪图片必须保留：
  `assets/images/hero/ys-premium-home-hero-v1.jpg`
- 未经用户明确授权，不得删除、移动、重命名、压缩或覆盖该图片。
- 图片优化不得导致文字、设备细节、人物或项目事实失真。

## 12. Git保护

- 每次修改前先运行 `git status --short`。
- 不得覆盖、回滚、清理或删除用户现有的未提交修改。
- 发现与当前任务无关的修改或未跟踪文件时，应保留并在最终报告中说明。
- 未经用户明确要求，不得执行：
  - `git reset --hard`
  - `git clean`
  - 强制检出覆盖文件
  - 自动提交
  - 自动推送
  - 合并分支
  - 删除分支
- 不得把历史压缩包、认证信息或无关大文件加入Git。

## 13. 修改流程

开始修改前必须：

1. 明确本次任务目标和验收标准；
2. 检查相关页面、样式、图片和资料来源；
3. 检查Git状态；
4. 明确哪些文件需要修改；
5. 只进行完成任务所需的最小改动。

用户要求“审核、检查、分析、评估”时，默认只读，除非用户明确授权修改。

## 14. 验证要求

由于当前项目没有正式构建命令，每次网站修改后至少完成以下检查：

1. 检查所有实际修改文件；
2. 检查HTML基本结构和标签闭合；
3. 检查CSS语法和明显冲突；
4. 检查修改页面的内部链接和图片路径；
5. 检查主要导航和页脚；
6. 检查桌面端显示；
7. 检查手机端显示；
8. 检查浏览器控制台错误；
9. 检查相关结构化数据；
10. 再次运行 `git status --short`。

- 可以使用安全的本地静态服务器预览网站。
- 启动临时服务器后，任务结束时应将其正常关闭。
- 没有实际检查的内容不得写成“已经验证”。
- 暂时无法检查时，应明确说明尚未验证的项目和原因。

## 15. Vercel和正式发布

- 未经用户明确授权，不得部署到Vercel。
- 未经明确授权，不得修改：
  - 正式域名；
  - DNS；
  - Vercel项目绑定；
  - 重定向；
  - Canonical；
  - robots.txt；
  - sitemap.xml；
  - 环境变量；
  - 生产环境设置。
- 用户授权部署时，应先完成本地检查，再说明准备部署的文件和风险。
- 部署完成后必须检查正式网站状态、主要页面、手机端、链接、图片和控制台。
- 不得把“文件已修改”表述成“网站已上线”。

## 16. 完成报告

每次任务完成后，应使用简体中文汇报：

1. 本次任务结论；
2. 修改了哪些文件；
3. 每个文件修改了什么；
4. 使用了哪些资料来源；
5. 执行了哪些检查；
6. 检查结果；
7. 未解决或尚未验证的问题；
8. 是否涉及部署；
9. 是否需要用户确认下一步。
